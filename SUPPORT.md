# Support And Compatibility Policy

Linketry is a free, self-hosted, pre-1.0 project. Community support is best effort and does not include an uptime, response-time, migration, or recovery SLA.

## Supported Release Line

The latest release published from `main` is the supported release line. Security fixes target that release. When requesting help, reproduce the problem on the latest release or explain why upgrading is not yet safe.

Linketry uses semantic versioning with this pre-1.0 policy:

- Patch releases (`0.x.y`) preserve documented `/api/v1` contracts and existing D1 data while delivering fixes, tests, documentation, and compatible enhancements.
- Minor releases (`0.x.0`) may intentionally change a pre-1.0 API, configuration, or workflow contract. Breaking changes must be called out in `CHANGELOG.md` with an upgrade path.
- Applied migrations are forward-only. They are never silently rewritten or rolled back by a normal deployment.
- The unversioned `/api/*` alias is deprecated compatibility behavior. New clients must use `/api/v1`.

## Supported Toolchain

| Component | Supported range |
| --- | --- |
| Node.js | 24.x |
| npm | 10 or newer |
| Wrangler | 4.111.0 or newer within major version 4 |
| Cloudflare Workers runtime | The compatibility date generated by the maintained deployment workflows |
| Admin browsers | Current stable Chrome, Edge, Firefox, and Safari; automated release coverage uses Chromium |

Use the repository lockfile with `npm ci` for deployments. Other runtimes, package managers, Wrangler major versions, modified migrations, and unreviewed Cloudflare bindings are outside the maintained deployment contract.

## Backup Expectations

The instance owner is responsible for data durability.

Before an upgrade:

1. Export a Linketry backup and record its creation time.
2. Prefer an R2 backup when the optional binding is enabled; otherwise retain a downloaded backup outside Cloudflare.
3. Verify the backup belongs to the intended instance and can be parsed by the restore preview.
4. Record the running version, target version, Git commit, migration digest, Worker/Admin domains, and D1/KV resource bindings.
5. Keep at least one previous known-good backup until redirects and Admin reads are verified after the upgrade.

Backups can contain destinations, configuration, and operational metadata. Store them as sensitive data and do not attach them to public support requests.

## Upgrade Contract

Use the protected GitHub Actions workflow described in [DEPLOYMENT.md](DEPLOYMENT.md). The release, commit, migration digest, backup reference, migration review, and target confirmation gates must pass before Cloudflare writes.

After deployment, verify:

- `/health` reports the intended version;
- Admin reports the same version;
- an existing short link redirects correctly;
- a disabled link remains non-redirecting;
- authenticated Links, Analytics, Audit, and backup views load;
- no production resource was replaced by a Demo resource.

Do not treat a successful build as proof that a Worker, Pages project, custom domain, or migration is live.

## Minimum Rollback Procedure

1. Stop additional deployments and preserve the failed run logs.
2. If no migration was applied, redeploy the last known-good commit through the same protected workflow and verify Worker/Admin version parity.
3. If a migration or data mutation was applied, do not run an older binary against an assumed older schema. Use the restore preview, restore the verified pre-upgrade backup when required, then deploy the matching known-good release.
4. Recheck `/health`, one existing redirect, one disabled link, Admin login, Links, and Analytics.
5. Rotate any credential exposed during diagnosis and invalidate copied logs or artifacts that contained it.

D1 is the source of truth. Rollback must preserve that ownership. Deleting KV can force safe cache repopulation; replacing D1 with KV data is never a recovery method.

## Requesting Help

Use [GitHub Issues](https://github.com/everett7623/Linketry/issues) for reproducible bugs and documentation problems. Search existing issues and [docs/KNOWN_ISSUES.md](docs/KNOWN_ISSUES.md) first.

Include the Linketry version, deployment track, browser or Node version, redacted error, expected behavior, actual behavior, and minimal reproduction. Do not include credentials, D1/R2 exports, visitor identifiers, or private domain inventories.

For an unpatched security issue, follow [SECURITY.md](SECURITY.md) instead of opening a public issue.

## Scope Limits

Best-effort project support covers the unmodified repository, documented Cloudflare architecture, and maintained deployment workflows. Custom forks, third-party proxies, unsupported runtimes, provider outages, DNS ownership disputes, custom migrations, and recovery from missing backups may require independent investigation.
